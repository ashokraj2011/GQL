How to read it
  • Four horizontal bands, top to bottom: each band is a product layer, always present.
  • Layer 2 is split: agents and artifacts are peers. Agents produce artifacts; artifacts constrain agents via input/output contracts.
  • Cross-cutting column on the right: seven concerns that aren't layers of their own — they attach at every layer.
  • Colors encode the layer: blue (anchor) · green (producing) · amber (orchestration) · red (proof) · purple (cross-cutting).

Six architectural claims this diagram bakes in
  1. The ledger is the single source of truth. Every consumer surface derives its view from the same append-only log. No parallel audit DB.
  2. DSSE is not optional. Unsigned evidence cannot be written to the ledger. Signature + keyId + algorithm live next to the payload.
  3. Links make the graph. links[] references upstream build attestations (SLSA, GitHub), sibling evidence, and prior reveal events. Turns a bag of packets into an attestation graph.
  4. Privacy-aware by default. Stored payload is { fieldHash, encryptedRef, identityDid, policyOutcome } — not raw prompts/responses.
  5. Reveal is audited too. A break-glass decrypt emits its own attestation (note the loop back to Assemble). Compliance can always answer "who looked at what, when, why."
  6. Consumers are opinionated. Each of the five surfaces is purpose-built for one buyer: operators (UI), compliance (17a-4), model-risk (MRM), SRE (incident), eng managers (daily control).
Compliance mapping — one arrow per framework
  • SR 11-7 → MRM Validation surface (policy trip counts, sample tests for validators).
  • SEC 17a-4 (2022 amendments) → 17a-4 Export surface (records + audit trail in human-readable form, prompt export package).
  • SOX → Separation-of-Duties column on the Daily-Control dashboards.
  • DORA → Incident Link + change-to-incident lineage graph.
  • SLSA / in-toto → links[] references in step 4 of the pipeline.
